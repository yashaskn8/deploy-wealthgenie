# ADR-008: Synthetic Data Limitations and Real-World Gap

## Status
Accepted

## Date
2026-07-23

## Context
WealthGenie's current recommendation model is trained on 20,000 synthetic investor profiles generated by `generate_correlated_dataset()`. While the supervisory target labels for these profiles are grounded in realized historical AMFI NAV performance statistics (`market_performance.csv`), the underlying investor profile features (Age, Income, Debt, Dependents, Emergency Fund, Horizon) are synthetically sampled using parametric statistical distributions (log-normal, Poisson, Gaussian).

To maintain scientific integrity, the boundary between a research prototype / synthetic benchmark and a production financial advisory system must be explicitly documented.

## Decision
We explicitly document the synthetic data limitations in `model_card.md` and establish an architectural migration path for real-world deployment:
1. **Documented Limitations:**
   - **Synthetic Demographic Distributions:** Synthetic profiles approximate real-world Indian investor correlations (e.g. income peaking in 40s, debt rising with dependents), but cannot capture unobserved macro-economic shocks, behavioral biases, or tail-risk distributions.
   - **Absence of Real Portfolio Outcomes:** Supervisory targets reflect historically constructed suitability policy rules, not empirical investor transaction histories or long-term portfolio performance outcomes.
2. **Future Migration Strategy:**
   - Designed modular feature ingestion schemas (`schemas.py`) so that the microservice can ingest real anonymized investor transaction logs and portfolio holdings without breaking existing API contracts.

## Alternatives Considered

### 1. Claim Production Readiness Without Real-World Validation
- **Pros:** Sounds impressive in project presentations.
- **Cons:** Scientifically dishonest; masks the fundamental gap between synthetic data and real-world financial behavior.
- **Reason for Rejection:** Fails scientific integrity standards.

### 2. Delay Pipeline Implementation Until Real User Data Is Available
- **Pros:** Avoids synthetic data entirely.
- **Cons:** Prevents building, testing, and validating end-to-end ML infrastructure, microservices, and explainability components.
- **Reason for Rejection:** Impractical for initial system development.

## Consequences

### Positive
- Transparent disclosure of research boundaries in model cards and system documentation.
- Modular pipeline design ensures smooth future transition when real user transaction datasets become available.

### Negative / Trade-Offs
- Reported model accuracy ($95.53\%$) reflects performance on synthetic profile distributions and cannot guarantee identical real-world generalization.
